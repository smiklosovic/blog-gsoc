And what about the release?

2013-08-20 by Stefan Miklosovic

This is the fifth one!

Multiple deployment

This is the big issue for Droidium. Lastly, I wrote about the possibility to be able to deploy more than more @Deployment. This is quite useful scenario when you need to get some libraries (in context of Android, not ShrinkWrap) along the test itself. You have the possibility to install some service packages as well. You just need to tell Droidium which @Deployment will be instrumented by @Instrumentable annotation placed on the deployment method. You can test multiple deployments on this demo. I am personally quite satisfied with the test coverage of the multiple deployment feature and after writing that I am quite confident in the event model of Arquillian itself.

Archetypes

I coded some very basic tests as archetypes a user can get generated in order to bootstrap his mobile testing experience. Archetypes are covering native and web testing scenarios as well. All is bundled in archetypes themselves so just generate that and you are done! Archetypes can be found here.

Headless mode

I am seeing a lot of possibilities in the headless testing. You could fully use the power of Maven and Arquillian (Droidium) to test your mobile applications in headless mode in a cloud, even doing functional testing. Right now it is possible to test headlessly without the issues (via -Pheadless profile in all tests in test/ dir) but this one reported by me. Screenshots are ugly so you can not take screenshots so conveniently to keep track of what was tested and how when doing it headlessly. It seems it is a bug.

APK builder

This project is postponed for a while. The guy behind this is my colleague Tadeas Kriz who has more important things to do right now and I heard there are some problems with Android APK integration with ShrinkWrap (problems on Android side) which are pending to be resolved. This one seems to be more time consuming than it appeared, regretfully.

Arquillian and JBoss Forge

I was hacking around JBoss Forge for a while and that resulted in the merged Arquillian Droidium extension for Arquillian Forge. The native and web plugin got merged, I did PR about container itself as well but it seems nobody cares :). It is more likely it gets passed once container itself is in Maven Central. I have to say it does not make much sense to have it there when you have to build that container locally on your own.

Screenshooter

There were plans to do some unified screenshooter for Arquillian but it seems to me this topic is out of scope right now and it is not actually so needed. Screenshooter for Droidium is implemented as a standalone Arquillian resource injected into the test which is bound to Android device instance. Screenshooter can take images of various image formats and they can be put in arbitrary directories under chosen names. That is pretty all what is needed to take it conveniently according to your needs. Check it here!

Event model

In order to provide easier entry point for beginners, I created simple event model images which keep track of what is going on behind the curtains. These are placed here.

Releasing Droidium bits

There is the JIRA task to release Droidium 1.0.0.Alpha1 this week but we are forced to postpone it for a while. The reason behind this is that Selendroid server of version 0.5.0 will be released in two weeks and I would love to base Droidium on it since, according to Dominik Dary from Selendroid team, it will be more stable and feature complete from testing point of view and it would be nice just to wait.

Another bug is related to Android ddmlib from Google. I found the regression which I submitted. This one was confirmed and repaired in the upstream and it will be present in the next version of ddmlib. The 1.0.0.Alpha1 should use the new version of ddmlib library without this issue.

More importantly, while figuring out the multiple deployment support, I submitted the PR which will be included in 0.5.0 enabling multiple instrumentation, theoretically. Up to this point, it was only possible to hook Selendroid server to one port, 8080, so another server was blocked from the operating. Right now you can choose your port on Android device side arbitrarily so more than one application could get instrumented. The logic behind this is deadly simple. When you have multiple deployments, you need to say which one of them will be instrumented while another will be just installed, serving as a service or resource package as a dependency. This is done with @Instrumentable. When you have multiple instrumentations, you have to somehow recognize which application will be instrumented by which Selendroid server, shifting this problem one level up. The preliminear answer to this is that it will be done via configuration you pass to that @Instrumentable, like @Instrumentable(viaPort = 8081). Once parsed, Selendroid server(s) will be hooked to these ports and you hook your WebDriver instances injected from Drone into the test as well just to them. With this, you can even test multiple Android applications simultaneously with multiple deployments. I am considering this as the final destination where I want to be with the deployment possibilities.

Conclusion

I am looking forward to get it into Maven Central so it will be (finally) publicly available. After multiple deployment feature which is currently under development (native extension is refactored because of that quite heavily), I would like to help in Aearogear project where I would bring Droidium on the scene. We are doing some testing on Aerogear as well and some areas are not covered. Droidium should fill this gap. The next thing is to polish the code as such, mostly unify the logging. I strongly believe that Droidium will be released before the end of GSoC.

I have to say that it is actually quite fun and I love doing this in its entirety. The one big lesson I got while developing this is that no matter how safe time plans you have, almost everything will take you more time you though it would and everything is more complicated at it seems for the first time. Secondly, every action takes some reaction and more open you are more good things will happen to you. Last but not least, it seems to me open source environment just changes everything and the openness is the thing which drives us all. The more I give, the more I get back.
